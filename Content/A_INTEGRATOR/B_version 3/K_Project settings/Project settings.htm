Overview

The project settings consist of a series of components that determine the attributes and building blocks that make up a project.

System administrators and project managers are responsible for defining project settings, and by default, only they can make changes to the them. Other users can view those settings for reference.

You will find the project settings towards the bottom of the list of items in the menu.

Units

This is the list of units of measurement used throughout the project, as defined by the project manager.

When creating an import template, users are asked to indicate the unit of measurement of each property. To do so, they must select one unit in a list generated from this table. It is important that you add all necessary units to the table prior to data import.

Deciding on the data units to use in a project is crucial to ensure consistency and to avoid having values of a single property with different units. For example, avoid having stress values in kPa and others in MPa. Such a situation would limit data integration as the analysis tools would perceive ‘stress in kPa’ and ‘stress in MPa’ as different properties.

Units listed here do not restrict the units that you can use in any files, maps, or other documents imported in the data management system; they are only used for data imported in the database.

Creating units using the button at the top right of the Current units table, as well as editing and deleting them is only available to users with proper permission level.

It is not possible to delete a unit if data recorded as having this unit exists in the database.

Classifications

This component is for importing categorical data in any theme.

An example of a rock code classification and its categories or classes could be something like this:

  • Rock Code:
    • Dolomite
    • Limestone
    • Chert

Having set classifications in a project provides a data consistency check in the sense that entries must correspond exactly to the existing classes. In the example above, if a user would try to import a data file containing typos, or where someone might spell limestone in two words (lime stone) instead of one, the import process would halt and an error message would prompt you to correct the data entries that do not correspond exactly to the specified classes.

Having set classifications for a project ensures that all data entered in the database respects the same classes, making data consistent for analysis throughout the project.

This is also useful in that it prevents you from having to type each class name into your template.

To define a classification:

  1. In the menu, click project settings, then click Classifications.
  2. In the System classifications table, give a name and description to your classification. You can also enter a description.
  3. Click Add classification.
  4.  Define its classes, by:
    1. Clicking the Classification’s edit button (insert button)
    2. In the Class name field of the Classes for classification window, name the class.
    3. Click Add Class. It is also possible to set a colour for each class by clicking add colour, then setting the colour of your choice using the colour charts or by entering the RGB# value.

If you plan to add numerous classes and wish to avoid typing them out by hand you can use the option to import the list of classes from a CSV file. To do so, click Import from file in the Classes for classification window and follow the given instructions. The CSV file should contain no header, just the list of class names.

It is possible to generate supplementary classes directly from the import template. These extra classes will only apply to the data imported via their specific import template and will not be added to the list of classes of the selected project classifications.

Neighbourhoods

Neighbourhoods are user-defined volumes of space within the project. Each is composed of one or more 3D polygons, has a meaningful name, and represents a volume that users generally refer to by its name rather than by coordinates. For example, a Neighbourhood in a mine could represent a level, a stope, a drive, or a shaft area.

Neighbourhoods greatly facilitate spatial queries as users do not require knowing or entering the x, y, z coordinates of the regions they are interested in; they have the possibility to simply select neighbourhood names instead. This is particularly useful when a neighbourhood includes many polygons.

The system automatically and continuously tracks what data resides in each neighbourhood.

To create a Neighbourhood:

  1. In the menu, click Project settings, then click Neighbourhoods.
  2. In the Current neighbourhoods table, click Create neighbourhood.
  3. In the Create neighbourhood dialogue, give your neighbourhood a name. You can also provide an optional description.
  4. Click Add, then click Close.

You must add at least one volume to a neighbourhood for it to identify already-existing data. In the Current neighbourhoods table, click the number in the # Volumes column. The neighbourhood can contain many volumes. Here, you can click Add new volume, which will give you the option to either define a bounding box or a polygon with a thickness.

Alternatively, you can import a polygon from a CSV file to create a new neighbourhood. This is done by clicking Import neighbourhood in the current neighbourhoods table. Your CSV file must contain a list of x,y coordinates with a header. You will provide the min-max z extents in the form.

When a new volume is created or edited, the system scans the entire database to identify which 3D data resides in it. It may take a a few moments to complete this operation. Similarly, the data import process scans all new data to track in which neighbourhoods, if any, each point resides.

Neighbourhoods are composed of one or more 3D volumes with each volume being delimited by either an x, y, z bounding box or a polygon with a defined Z-axis range.

  • Bounding box: A cuboid defined by its south-western and north-eastern coordinates.
  • Polygon: Comprised of an unlimited number of XY points that join to create a closed 2D polygon, where the last point joins with the first to complete the polygon. To turn the 2D polygon into a 3D volume, you must define maximum and minimum Z-axis values.
Neighbourhoods are independent from one another; they can overlap. Therefore, any point can belong to many neighbourhoods.

Mine levels

This component is for defining the minimum and maximum elevation for each mine level of the project. You can add and delete mine levels from this setting depending on user permissions. The Mine levels are only used in the Mine model compilation and Geohazard reports section of .

Shifts

Shifts are set portions of time within a 24-hour period, allowing you to set active time slots for a project. Shifts facilitate time-based queries as users do not need to enter specific start and end time values. Instead, they can simply select specific shift names.

A typical example is to set a morning, an afternoon, and a night shift corresponding to the underground operations schedule.

We recommend that shifts do not overlap, however, does not limit or control this. You can edit shift names, descriptions, and start time and end time values. Ability to create, edit and delete shifts is permission-based.

To set shifts in a project:

  1. In the menu, click Project settings, then click Shifts.
  2. In the Current shifts table, click Add new shift.
  3. In the Add new shift dialogue box:
    1. Give the shift a name and description.
    2. Define the start and end times.
  4. Click Add, then click Close.

Data set metadata

This component allows you to create, edit, and delete metadata related to data sets in projects.

To add new metadata:

  1. Within the Project settings, click Data set metadata.
  2. Click Add new metadata field.
  3. In the Add new metadata dialogue box, give the metadata a name, a unit, and a type.
  4. Click Save.

Along with a list of your metadata, you will be able to edit all parameters as well as delete entries within the Data set metadata table.

If a data set has populated a metadata field, you cannot modify its unit and type.

If you do not have permission to edit Data set metadata, this page will only contain a table displaying the items and their settings.

File types

This component is a list of file types available for use as a parameter in the File manager and Maps / plans / sections pages. From here you can add, edit, and delete file types if you have proper permission level.

Sample media

This component contains a table listing all of the sample media available for samples in the project. Project managers and system administrators can add new sample media (button at top right), as well as edit and delete existing ones.

Data sets

This component allows administrators and project managers to create, edit, and delete data sets in projects.

To add a new data set:

  1. Within the Project settings, click Data sets.
  2. In the Data sets table. click the Create button.
  3. In the Add data set dialogue box, give the data set a name and a description (optional).
  4. Click Save.

You can edit and delete entries within the Data set table.

Import templates

This component allows you to edit a template's name or NDV, as well as delete or deactivate them if you have the proper permission level.

It is not possible to delete a template that has been used to import currently-existing data. See the column Is data? to know if it is the case. You can either remove all the batches that used this template (Is data? will become "no") and then you can delete it. Or you click the Status Activeto toggle it to Inactive, which will hide that template from the list of templates available for import.

Groups

This component allows you to edit the Group name and Description if you have the proper permission level.

To do so, make sure you are in a theme that allows group names. Because group names belong to data sets, you have to select a data set from the second drop-down menu in order to see a table with its group names.

The only way to delete group names is to delete the whole data set from the Data sets  page of the Project settings (see "Data sets" on page 1).

Assigned users

Project managers or system administrators can manage the user roles within a specific project. This page contains a list of user accounts for the current project, including user name, email address, organization, and assigned role. System administrators and project managers can remove users by clicking the red X at the end of their row, or add users to the project by selecting their user name from the drop down list at the bottom of the table and selecting a role for them, then clicking Save.

Permissions are given according to the role assigned. Roles are set per project, so a user can have a different role (access level) per project.

To modify a user's role, you need to remove their current role and assign them a new one.
Users must already exist in the system before you can assign them a user role in a project. Adding users to the system is limited to system administrators.
